# 2026-05-27 工作日志

## SDK 元素属性方法改为本地缓存
- `isVisible()`, `isOffscreen()`, `isEnabled()`, `bounds()`, `isCheckable()`, `isChecked()`, `isClickable()`, `isScrollable()`, `isSelected()` 全部改为使用 `this.info` 本地属性，不再发起网络请求
- 用户需先 `await el.refresh()` 获取最新状态

## SDK xpath 属性简化
- `xpath` 从 `string & function` 简化为 `readonly string`，删除 `xpathOf()` 方法
- 重命名为 `selector`，与 `windowSelector`/`listSelector` 风格一致

## SDK elementSelector → listSelector 重命名
- Element 类的 `elementSelector` 属性重命名为 `listSelector`
- 涉及 element.ts, types.ts, flow.ts, tests
- 后端 API 响应字段名 `elementSelector` 保持不变（协议层），SDK 内部映射为 `listSelector`

## findOne / findFirst / find 语义拆分
- 后端: `ElementResponse` 新增 `total` 字段，`/api/element` 返回匹配元素总数
- SDK `findOne(xpath)`: 匹配多个时报错，期望恰好 1 个
- SDK `findFirst(xpath)`: 匹配多个也不报错，返回第一个
- SDK `find(xpath)`: `findOne` 的别名（deprecated）
- `waitFor` 内部改用 `findFirst`，`findAll` 内部也改用 `findFirst`
- 修复 element.ts 中残留的 `response.elementSelector` → `response.listSelector`

## scrollIntoView wait 选择器修复
- 问题：`scrollIntoView` 使用 `this.listSelector` 作为 `wait` 参数，但 listSelector 是通用 XPath（如 `Button[@FrameworkId='Chrome']`），匹配所有同类元素
- 后端 `is_offscreen` 只检查第一个匹配元素，当第一个可见但目标元素在屏幕外时，后端误判为已可见而停止滚动
- 修复：将 `wait: this.listSelector` 改为 `wait: this.buildXpathFromProps(propNames)`，生成包含 `@Name` 等唯一属性的 XPath，确保后端只匹配目标元素

## scroll API window 参数 + buildXpathFromProps 去重
- 问题1：后端 scroll API 硬编码 `"Window"` 选择器遍历全部29个窗口，每次 wait 检查耗时数百毫秒
- 修复1：`MouseScrollRequest` 增加 `window: Option<String>` 字段，scroll_mouse 使用该选择器限定搜索范围
- 修复1续：SDK `scrollMouse` 增加 `window` 参数，`scrollIntoView` 传递 `this.windowSelector`
- 问题2：`buildXpathFromProps` 在 `listSelector` 已有 `@FrameworkId='Chrome'` 时又追加 `@FrameworkId='Chrome'`，导致重复谓词
- 修复2：新增 `parseExistingAttrs()` 方法解析 listSelector 最后一步已有属性，`buildXpathFromProps` 跳过已存在的属性
